# OmniScribe — AI Workflow Rules

> Rules for any AI coding agent (Claude Code, Cursor, Codex, Copilot, future) working on this build. Commands, not suggestions. Medical software has regulatory stakes; undisciplined AI changes can hurt patients, cost licenses, and break trust.

## Approach

Build OmniScribe incrementally using a **spec-driven greenfield workflow**.

- The six context files (`project-overview`, `architecture`, `ui-context`, `code-standards`, `ai-workflow-rules`, `progress-tracker`) define the system.
- The journey files in `journeys/` describe the user experience to build toward.
- The spec files in `context/specs/` define each unit of work.
- The agent **executes specs**. It does not invent product behavior, architectural decisions, or visual decisions that aren't in the context files or journeys.
- When something is undefined, the agent stops and asks — it does not guess.

## Scoping Rules

1. **One unit at a time.** A unit is one entry in `context/specs/00-build-plan.md`. Finish it (verified) before starting the next.
2. **Prefer small, verifiable increments over large speculative changes.** A 4-file PR is better than a 40-file PR.
3. **Do not combine unrelated system boundaries in one implementation step.** UI + DB migrations + workers in one PR breeds hidden defects. Split.
4. **Do not start new work mid-stream.** If you're 80% through Unit 03 and notice Unit 05 has a related issue, log it in `progress-tracker.md` open questions — don't fix it now.
5. **Match scope to the ask.** Bug fix = fix the bug. Refactors get their own units.

## When to Split Work

Split if a step combines any of:

- UI + background-task changes in the same PR.
- Two or more unrelated API routes.
- Behavior that isn't clearly defined in the context files or journeys (you'll end up inventing — stop and define first).
- More than ~6 modified files where changes don't share a single concern.
- Anything that can't be verified end-to-end inside one development session.

## Handling Missing or Ambiguous Requirements

1. **Do not invent product behavior.** If `project-overview.md` or a journey doesn't say it, you don't build it.
2. **Do not invent architecture.** If `architecture.md` doesn't define where this lives, you don't decide alone.
3. **Do not invent visual decisions.** If `ui-context.md` doesn't have a token or pattern, you don't make one up — propose it as a token addition first.
4. **If a requirement is ambiguous**, resolve it in the relevant context file before implementing. Update the file, then build against the updated text.
5. **If a requirement is missing**, add it to the **Open Questions** section of `progress-tracker.md` and pause. Ask the user.
6. **If the user gives a vague direction** ("make capture better"), ask which aspect, which file, and what "better" means in measurable terms.

## Three-Lens Evaluation (mandatory, every feature)

**Clinician** — would a licensed provider trust this with their license?
- Does the language match how clinicians document?
- Does the workflow respect how clinicians work?
- Is the clinician in control of every clinical decision?

**Medicare Compliance Officer** — would this establish medical necessity + skilled care and survive a MAC audit?
- Objective measures captured?
- Skilled intervention documented as such?
- Signature + attestation correct?

**Insurance Auditor** — could an auditor reconstruct this visit?
- Provenance traceable on every fact?
- AI-generated vs clinician-edited distinguishable?
- Audit log complete + PHI-free?

If any lens fails, the feature is not done. Document the evaluation in the PR description.

## Protected Files (do not modify without explicit instruction)

- **`src/components/ui/*`** — shadcn-generated primitives. Extend with CVA variants in sibling files; do not edit the primitive itself.
- **Applied migrations in `prisma/migrations/*`** — never edit; write a new one.
- **`src/generated/prisma/*`** — generated client; regenerated by `prisma generate`.
- **`finalJson` write paths** — only the sign route writes it. Anywhere else is a bug.
- **`src/services/llm/` PHI guard (`assertProviderAllowedForPHI`)** — never weaken; never bypass.
- **Audio S3 deletion paths** — there are none, and there should be none.
- **Anti-regression rules in `context/architecture.md` Invariants** — these are the law; never delete or weaken.
- **Journey files in `journeys/`** — describe the target experience; if the implementation diverges, the docs need updating, not the journey rewritten silently.

## Keep Documentation in Sync

After any implementation change affecting:

- **System architecture or boundaries** → update `context/architecture.md` in the same PR.
- **Storage model** → `context/architecture.md` Storage Model section.
- **Code conventions** → `context/code-standards.md`.
- **Feature scope** → `context/project-overview.md`.
- **Visual tokens or patterns** → `context/ui-context.md`.
- **User-experience flow** → relevant `journeys/NN-*.md`.
- **Phase / unit progress** → `context/progress-tracker.md` (always, after every meaningful change).
- **New invariant** → add to rules list in `context/architecture.md`.

Docs out of sync = next agent invents from scratch = drift = rework.

## Verification Before Moving to the Next Unit

All of these must be true:

1. Current unit works end-to-end within its defined scope (manually verified).
2. No invariant in `context/architecture.md` was violated (self-audit against the 24-rule list).
3. 3-tap UX test passes on any clinical surface touched.
4. Three-lens evaluation passes (Clinician / Compliance / Auditor) — documented in PR.
5. `context/progress-tracker.md` updated (unit moved to Completed).
6. `npm run build` passes locally.
7. `npm run lint` passes locally.
8. Relevant tests pass; new tests exist for new behavior.
9. `npm run dev:workers` was running while testing any flow that ends in a generated note.
10. Audit log writes exist for every PHI mutation in new code.
11. No new hardcoded status colors, no native `confirm()`, no `text-[Npx]` in clinical/admin surfaces.
12. Any new env vars documented in `context/architecture.md` Environment Variables.

## How to Resume After a Break

Read in this order:

1. **`CLAUDE.md`** (or `AGENTS.md`) — entry-point reminder.
2. **`journeys/02-typical-visit.md`** — anchor your mental model in the user experience.
3. **`context/project-overview.md`** — what + scope.
4. **`context/architecture.md`** — how it's built + invariants.
5. **`context/ui-context.md`** — visual language.
6. **`context/code-standards.md`** — conventions.
7. **`context/ai-workflow-rules.md`** — this file.
8. **`context/progress-tracker.md`** — current state, in-progress unit, open questions.
9. **`context/specs/NN-<current-unit>.md`** — the unit you're working on.
10. **Any `journeys/NN-*.md`** that exercise the surface you're building.

If `progress-tracker.md` shows a unit "In Progress," continue it. If it shows the prior unit "Completed," pick up the next per `00-build-plan.md`. If you can't tell what state you're in, ask the user — do not guess.

## Common Failure Modes (recognize and avoid)

- **"Let me refactor while I'm here."** → No. Separate PR.
- **"I'll skip the audit log for this one read — it's just a list."** → No. PHI reads are logged.
- **"I'll mock the LLM for this PHI test."** → Don't run PHI through LLM in tests at all; use deterministic fixtures.
- **"I'll add a default to this switch on `NoteStatus`."** → No. Use `assertNever`. Default hides bugs.
- **"I'll silently fall back to OpenAI if Bedrock fails."** → No. OpenAI is not PHI-allowlisted. Fail loud.
- **"I'll spawn a second worker fleet for safety."** → No. One fleet per Redis.
- **"I'll just modify this migration to fix it."** → No. Migrations are append-only.
- **"I'll bypass the PHI guard for this performance test."** → No. The guard is the law.
- **"I'll use `confirm()` for the leave prompt."** → No. AlertDialog.
- **"I'll skip the three-lens evaluation — it's just a UI tweak."** → No. Every PR.

## Asking the User

When you must stop and ask:

- **Be specific.** "Should X be opt-in or default-on?" beats "How should we handle X?"
- **Offer options.** Use `AskUserQuestion` when there are 2–4 clear paths.
- **Cite the source of the ambiguity.** "Spec 03 says X but journey 02 implies Y — which is canonical?"
- **Don't ask permission for safe, reversible work.** Asking "may I edit the file?" wastes both of your time. Ask when the action is risky, irreversible, or ambiguous.

## Tooling Norms (Claude Code-specific)

- **Use Read for known paths**, grep via Bash for known strings, Explore agent for open-ended search.
- **Parallel tool calls** for independent work.
- **TodoWrite** for any task with ≥ 3 sub-steps.
- **Mark chapter** when shifting phase.
- **Never `--no-verify` / `--no-gpg-sign`.** Hooks catch what you missed.
- **Never amend the previous commit** to "fix" a failure — create a new commit.

## When You Are Done with a Unit

1. Verification checklist (above) all green.
2. PR open with title `feat(unit-NN): <slug>`, description includes what / why / verify / three-lens, link to spec.
3. `context/progress-tracker.md` shows unit "Completed" with date + sources cited.
4. Any new open questions logged in `progress-tracker.md`.
5. Next unit's spec exists (if it didn't) — or, if all spec'd units done, propose the next units to spec.

Then — and only then — start the next unit.
